# 2026-06-19 개발로그

## Render/Supabase 환경 API PostgreSQL region lookup 보정

### 요청/배경

- Render 프론트 배포 후 Home 화면의 Fine Dust 값이 `10`으로 표시됨.
- live 환경 API 확인 결과 실제 미세먼지 값이 10인 것이 아니라, 환경 관측값이 `null`로 내려와 프론트 fallback 기본값 10이 표시되고 있었음.
- `GET /api/v1/environment/current?region=India&city=Hyderabad` 응답에서 `mode=fallback_cache`, `aqi=null`, `pm25=null`, `pm10=null`을 확인함.

### 원인

- 환경 API refresh 중 `create_environment_observation -> resolve_region_id` 경로에서 PostgreSQL/Supabase가 아래 SQL 조건의 파라미터 타입을 추론하지 못함.

```text
(? IS NULL OR city = ?)
```

- Render live 오류:

```text
psycopg.errors.IndeterminateDatatype
could not determine data type of parameter
```

### 수정 파일

```text
backend/app/repositories/sqlalchemy_repositories.py
backend/tests/test_environment_data_adapter.py
docs/ar/CareShot_AR_전체개발순서.md
docs/ar/development_logs/2026-06-19_개발로그.md
```

### 구현 내용

- `resolve_region_id`에서 city 조건을 nullable predicate로 처리하지 않도록 변경함.
- city가 있으면 SQL에 `AND city = ?`를 직접 추가함.
- city가 없으면 city 조건을 붙이지 않음.
- PostgreSQL/Supabase에서 타입 추론이 필요한 `? IS NULL` 조건을 제거함.

## Home 환경 정보/Care Risk Score 초기 표시 지연 보정

### 요청/배경

- 사용자가 Home 화면에서 환경 정보와 Care Risk Score 정보가 늦게 반영된다고 지적함.
- 확인 결과 API 응답 자체가 늦게 올 수는 있지만, 더 큰 문제는 프론트가 응답 전까지 데모 기본값을 먼저 표시하는 구조였음.
- 기존 초기 표시값:

```text
location: New Delhi 고정 분기
temperature: 24
humidity: 56
aqi: 10
pm25/pm10: 10
care risk score: devices[0].score = 82
```

### 수정 파일

```text
frontend/src/app/pages/Home.tsx
docs/ar/CareShot_AR_전체개발순서.md
docs/ar/development_logs/2026-06-19_개발로그.md
```

### 구현 내용

- `getUserProfile` 이후 `evaluateCareRisk`와 `getCurrentEnvironment`를 `Promise.all`로 묶지 않고 독립 호출로 분리함.
- `isCareRiskLoading`, `isEnvironmentLoading`, `profileLocation` 상태를 추가함.
- Care Risk 응답 전에는 82점 같은 데모 점수를 보여주지 않고 `Updating` 상태를 표시함.
- 환경 API 응답 전에는 `24`, `56`, `10` 같은 fallback 숫자 대신 `...` 또는 `--`를 표시함.
- location label을 New Delhi/Gujarat 하드코딩 분기에서 API observation 또는 사용자 profile region/city 기반 표시로 변경함.
- Care Risk API와 환경 API 중 하나가 늦어져도 다른 카드의 갱신이 같이 막히지 않도록 로딩 상태를 분리함.


결과:

```text

```

재검증:

```powershell

```

결과:

```text

```

### 실패/수정 사항

- 최초 빌드는 코드 오류가 아니라 `esbuild` 실행 권한 문제(`spawn EPERM`)로 실패함.
- 권한 상승 후 동일 명령을 재실행하여 Vite production build 성공을 확인함.
- 빌드 중 chunk size warning은 남아 있으나 기존 번들 크기 경고이며 이번 수정으로 인한 실패는 아님.

### 남은 작업

- 현재 프론트 Render Static Site가 `main` 브랜치를 바라보고 있으므로, 이번 수정이 `taehee` 브랜치에만 push되면 live 프론트에는 바로 반영되지 않을 수 있음.

```text
1. taehee -> main merge 후 Render auto deploy
2. 또는 Render Static Site Branch를 taehee로 변경 후 deploy
```
