## 22-9. Care Risk 환경 factor 응답/점수 반영 검증 및 보정

요청:

```text
environment_adapter.get_environment() 결과가 Care Risk API 응답과 score engine에 정상적으로 들어오게 고치고,
Ahmedabad/Gujarat mock 또는 live 환경값 중 습도/AQI/몬순 값이 있으면 trigger_reason[1], [2]까지 생기게 검증한다.
```

사전 확인:

```text
POST /api/v1/care/risk/evaluate
-> environment_observation 응답 포함
-> 최신 live/cache row 기준 humidity=71, AQI=68, monsoon=null
-> trigger_reason[0]: 최근 일평균 사용 시간은 6.0시간입니다.
-> trigger_reason[1]: 현재 습도는 71%로 높습니다.

AQI는 150 미만이라 제외, monsoon은 null이라 제외.
```

확인된 문제:

```text
ENVIRONMENT_OBSERVATION cache row는 monsoon을 top-level monsoon_intensity 필드로 보유한다.
하지만 CareRiskScoreEngine.environment_score()는 payload.rain_monsoon_intensity / payload.monsoon_intensity만 읽고 있었다.
따라서 DB cache row에서 monsoon_intensity가 있어도 몬순 factor가 빠질 수 있었다.
```

```text
02_데이터연동/db/careshot_ar_mock.db
```

```text

- environment.rain_monsoon_intensity
```

실패 및 수정:

```text

FastAPI 8791 프로세스를 재시작한 뒤 실제 HTTP 응답에서 trigger_reason[3]까지 확인했다.
```

남은 한계:

```text

실시간 provider가 AQI 150 미만 또는 monsoon null을 반환하면 해당 factor는 정상적으로 제외된다.
```

## 23-4. 냉방 multi-turn 통합 질문 반복 루프 수정

요청/배경:

```text
"냉방/기능이 잘 작동하지 않아요" 이후 고객이 "바람이 약하고 안시원해요"라고 답해도
챗봇이 동일한 통합 질문을 다시 반복했다.
사용자는 위험 신호가 없다고 말하지 않았지만, 이미 바람 약함/냉방 불량 상황을 답했으므로
같은 질문 전체를 반복하는 것은 UX상 무한 루프처럼 보인다.
```

원인:

```text
현재는 LLM API가 아니라 룰 기반 slot 추출 + LLMServiceMock 구조다.
`바람이 약하고 안시원해요`는 사람이 보기에는 냉방 상세 답변이지만,
기존 slot extractor는 이 문장을 `symptom_location`이나 `environment_context`로 채우지 못했다.
따라서 `missing_slots`가 `risk_signal,symptom_location,environment_context` 그대로 남아
프론트 호환 응답이 같은 통합 질문을 반복했다.
LLM 미연결도 표현 이해 한계의 한 원인이 될 수 있지만, 직접 원인은 현재 룰 기반 slot 추출 누락이다.
```

수정 파일:

```text

04_백엔드/tests/test_conversation_state_multiturn.py
00_계획수립/CareShot_AR_전체개발순서.md
```

수정 내용:

```text
- weak_airflow 수집 중 `바람`, `약`, `시원`, `냉방` 등 냉방 상세 표현이 들어오면
  `symptom_location=outlet`, `environment_context=unknown`으로 채움
- 그러면 남은 slot은 `risk_signal`만 되므로 같은 통합 질문을 반복하지 않음
- `risk_signal`만 남은 경우 질문 문구에 "없다면 '아니요'라고 답해주세요."를 추가
- `냉방/기능...` -> `바람이 약하고 안시원해요` 케이스 회귀 테스트 추가
```

