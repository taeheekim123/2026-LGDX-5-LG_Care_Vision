## 14. 예방 알림형 흐름 설계 보강

### 보강 사유

기존 설명은 사용자가 챗봇에 문의를 입력하는 `사용자 문의형 흐름` 중심이었다.

하지만 CareShot의 가전 관리 파트는 사용자가 묻기 전에 ThinQ 등록 가전 사용 로그와 인도 지역 기후/환경 데이터를 결합해 `Care Risk Score`를 산정하고, 기준 이상일 때 예방 관리 필요성을 알림 형태로 선제 제공해야 한다.

따라서 시스템 아키텍처와 전체 개발순서에 `예방 알림형 흐름`을 별도 진입점으로 추가했다.

### 반영 내용

- `CareShot_AR_시스템아키텍처.md`에 두 가지 진입 흐름을 추가했다.
  - 예방 알림형 흐름
  - 사용자 문의형 흐름
- 기존에 적었던 `PreventiveCareScheduler` 중심 표현은 정정했다.
- 예방 알림형은 단순 주기 실행이 아니라 `Care Risk Score` 기준으로 동작한다.
- `EnvironmentDataAdapter`를 통해 외부 환경 API를 실시간 또는 준실시간으로 조회하고, DB cache freshness를 확인하도록 설계했다.
- `CareRiskScoreEngine`을 백엔드/AI 서비스 구성요소로 추가했다.
- `PreventiveCareRecommendationEngine`, `CareContentMatcher`, `Care Risk / Alert DB` 흐름을 추가했다.
- 예방 알림형 sequence diagram을 추가했다.
- 사용자 문의형 sequence diagram과 구분했다.

## 15. Image-based AR 모델명/구조타입 확장 설계 보강

### 보강 사유

AR 방식은 객체인식 AR이 아니라 image-based reference overlay로 진행한다.

또한 ThinQ 앱에 등록된 제품을 대상으로 하므로 고객 기기의 `model_name`은 서비스 입력 단계에서 exact value로 들어올 수 있다. 따라서 모델이 다양해도 `model_name -> product_type -> structure_type -> reference_image -> part_map_version -> AR guide template` 구조로 매핑하면 된다.

## 23. 15단계 - PostgreSQL + pgvector 최종 DB 전환

### 사용자 작업 원칙 반영
