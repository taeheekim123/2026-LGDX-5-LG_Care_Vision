# 2026-06-09 개발로그

## 17.3 Device Care History 조회 View/API 구현 완료

목표:
- 가전 탭의 관리/A/S 내역을 프론트가 `content_view_logs`, `ar_session_logs`, `care_activity_logs`, `service_route_logs`, `expert_as_requests`에서 직접 조합하지 않도록 백엔드 통합 조회 API로 제공한다.
- `GET /api/v1/devices/{device_id}/care-history` 하나로 summary와 timeline items를 반환한다.

구현:
- `04_백엔드/app/routers/devices.py`
  - `GET /api/v1/devices/{device_id}/care-history` 추가
  - `user_id` 필수 query, `service_flow_type` optional filter, `limit` optional 지원
- `04_백엔드/app/schemas.py`
  - `DeviceCareSummary`
  - `CareHistoryItem`
  - `DeviceCareHistoryResponse`
- `04_백엔드/app/services.py`
  - `get_device_care_history()` 추가
  - summary fallback과 history item title/status/source id 정규화 추가
- `04_백엔드/app/repositories/sqlalchemy_repositories.py`
  - `get_device_care_history(user_id, device_id, service_flow_type=None, limit=50)` 추가
  - `care_activity_logs`를 우선 사용하고, 연결 활동이 없는 `content_view_logs` / `ar_session_logs`를 fallback 이력으로 통합
  - `ar_step_logs`를 AR session별로 집계해 `step_log_count`와 최신 step 시각을 AR Guide 이력에 반영
  - `service_route_logs`와 `expert_as_requests` 기반 expert A/S 이력 통합
- `04_백엔드/app/main.py`
  - devices router 등록
- `05_프론트엔드/types/careshot.ts`
  - 백엔드 응답과 맞도록 `DeviceCareSummary.summary_id` nullable 처리
  - `CareHistoryItem`의 `user_id`, `device_id` optional 처리
- `04_백엔드/tests/test_device_care_history.py`
  - 공식 콘텐츠 열람 완료 이력
- AR Guide 세션 이력
- AR step 로그 집계 반영
  - expert A/S route 이력
  - `device_care_summary` 반환
  - `service_flow_type` 필터
  - 최신순 정렬
  - 없는 device 404 처리 검증
